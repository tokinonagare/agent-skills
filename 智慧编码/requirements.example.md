# 开发需求

## ⚠️ 重要：AI 必须遵守的规则

**在开始开发前，请仔细阅读以下规则，这些是强制性要求：**

### 1. 分支信息记录（强制）
- 创建新分支后，**立即**更新下方"分支信息"部分的"分支名称"字段
- 分支名称必须使用有意义的英文，遵循 kebab-case 格式（例如：feature-user-auth）
- **不要**在主分支（master/main）直接提交代码

### 2. PR 链接记录（强制）
- 创建 PR 后，**立即**更新下方"分支信息"部分的"PR 链接"字段
- PR 链接格式：`https://github.com/owner/repo/pull/123`
- PR 标题和描述要清晰说明改动内容

### 3. 开发日志记录（强制）
- **每次**代码修改后，在"开发日志"部分追加新记录
- 记录格式：
  ```markdown
  ### [YYYY-MM-DD HH:MM:SS] - [修改原因]
  - 修改内容：[简要描述]
  - 涉及文件：[文件列表]
  - 测试状态：[是否通过]
  ```
- 修改原因包括：初始开发、PR 评论反馈、CI 失败修复、代码优化等

### 4. 代码规范（强制）
- 必须遵循项目的代码规范（参考 `/Users/tokinonagare/Work/kevinanew/auto_coding/prompt` 目录下的规范文件）
- 提交前运行 lint 和格式化工具
- 不要使用 `--no-verify` 跳过 Git hooks

### 5. 测试要求（强制）
- 所有新功能必须编写单元测试
- 测试覆盖率要达到要求
- 确保所有测试通过后再提交

## 附加说明

### 遇到问题时
1. 优先查阅项目已有代码，学习现有模式
2. 参考项目规范文档
3. 如果尝试 3 次仍未解决，记录问题并寻求帮助

### 代码提交流程
1. 开发功能 → 编写测试 → 确保测试通过
2. 运行 lint 和格式化工具
3. 提交代码（遵循 commit 规范）
4. 创建 PR（开启自动合并）
5. 等待 CI 通过
6. 监控 PR 评论，及时响应
