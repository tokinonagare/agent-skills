# Dedup Evidence Guide

## 目的

判断 open Issue 是否已经覆盖当前问题，避免重复提单，同时保留足够的证据补充空间。

## 查重顺序

1. 只看 `open` Issue
2. 对比标题、描述、复现步骤
3. 对比证据
4. 再决定是否跳过、补充或新开 Issue

## 证据相似度判断

### 高度相似，视为重复

满足多数项即可跳过：

- 页面或模块相同
- 复现路径相同
- 失败时机相同
- 截图展示的 UI 状态相同
- 录屏里的交互结果相同
- 控制台错误栈相同
- 网络失败的接口、状态码、响应内容相同

处理方式：

- 直接判定重复
- 不再提同一个问题
- 继续找下一个问题

### 现象相同，但证据不相似

可能出现：

- 同一功能错误，但当前截图显示了新的 UI 状态
- 同一报错，但当前录屏提供了新的操作路径
- 同一页面问题，但当前控制台栈或网络响应不同
- 同一问题在不同分辨率、不同登录态、不同浏览器下复现

处理方式：

- 把当前证据补充到已有 open Issue
- 在正文中注明新的复现条件或新的环境差异
- 如果无法补充现有 issue，则新开 issue，并注明与旧 issue 的区别

## 补充证据建议

- 新截图要标明当前环境
- 新录屏要包含完整操作链路
- 新控制台日志要保留完整栈
- 新网络信息要保留请求 URL、状态码、响应摘要

## 简单判断口径

- “同一个问题 + 同一批证据” -> 重复
- “同一个问题 + 新证据” -> 补充后再决定是否提单
- “类似现象但根因不同” -> 作为新问题处理
