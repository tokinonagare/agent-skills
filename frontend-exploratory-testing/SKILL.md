# Frontend Exploratory Testing

这个 skill 用于以前端工程师视角做探索性测试，目标不是“跑完脚本”，而是主动找出真实缺陷并形成可提交的 Issue。

## 适用场景

- 需要快速浏览并测试一个前端软件的主要功能
- 需要检查登录、表单、跳转、响应式布局、错误提示、控制台报错
- 需要把发现的问题整理成可直接提交到 Issue Tracker 的内容

## 核心原则

- **单次单任务**：每轮探索性测试**只寻找并报告一个**最具代表性或最严重的问题。一旦发现并验证一个问题，立即停止探索并进入提交流程。
- **必须提交 Issue**：任何发现的缺陷、UI 异常或报错，只要不属于已存在的重复 Issue，**必须**提交到 Issue Tracker。严禁只停留在本地记录或口头汇报。
- **系统化查重**：在提交 Issue 前，必须通过 `gh issue list` 检查 Open 状态的问题。不仅要看标题，更要对比复现手法、现象表现以及证据相似度。
- **证据严谨性**：发现问题后，必须先采集完整的证据（上下文截图、报错栈、录屏），再进行提交。
- **保持输出质量**：保持中文输出，Issue 内容要具体、可复现、可验证。

## 工作流

### 1. 制定测试计划

开始前先快速浏览页面和主要入口，明确本轮目标是**找出一个问题**。列出本轮重点区域，例如：

- 登录/注册流程
- 表单填写与提交
- 页面跳转与返回
- 空状态、错误状态、加载状态
- 响应式布局
- 控制台报错和网络失败

记录当前轮已经测试过的区域，避免重复覆盖。

### 2. 执行探索性测试

使用 `agent-browser` 与页面交互：

- 点击按钮、菜单、链接
- 输入正常值、边界值、非法值
- 切换页面、刷新、返回
- 缩放到不同分辨率或移动端宽度
- 观察 DOM、样式、遮挡、溢出、文案截断
- 查看控制台日志和网络请求失败

每次操作后都检查页面反馈是否符合预期，关注：

- UI 错位、遮挡、闪烁、溢出
- 点击无响应、状态未更新、数据不一致
- 错误提示缺失、文案不友好、恢复路径不清晰
- JS 报错、请求失败、白屏、卡死、崩溃

更完整的测试发散清单见 [exploration-playbook.md](references/exploration-playbook.md)。

### 3. 10 分钟换思路

如果在同一方向上持续约 10 分钟没有发现新问题，立即切换到另一个角度：

- 正常流程 -> 异常输入
- 桌面端 -> 移动端
- UI 检查 -> 网络层检查
- 单页检查 -> 跨页面流程
- 功能检查 -> 错误恢复与边界值

切换时记录原因和新的测试方向，避免在同一路径上反复耗时。

### 4. 发现问题后的证据采集

- **测试环境**：允许使用 `https://h5.shafayouxi.org/` 进行测试和截图。
- **证据采集**：普通问题优先截图，包含完整上下文；崩溃、白屏等优先录屏并保存控制台报错栈。证据必须足以复现问题。
- **强制上传与引用**：截图**必须**通过以下方式提交，严禁仅保留本地记录：
  1. **优先上传附件**：调用 `gh` 工具将图片作为附件上传到 Issue（若命令行工具不支持，转第 2 点）。
  2. **仓库内链接引用**：若无法上传附件，必须先将截图文件提交到仓库分支 `exploratory-testing-evidence`，然后获取 GitHub 的 Blob 视图链接（格式：`https://github.com/[OWNER]/[REPO]/blob/[BRANCH]/[FILE]`），并直接在 Issue 评论中贴出该链接。
  3. **禁止事项**：严禁使用 `raw.githubusercontent.com` 作为唯一证据链接，因为私有仓库无法公开访问该地址。

### 5. 查重与相似度判断

提交前，只检查 Issue Tracker 中 `open` 状态的 Issue。
- **一致性检查**：检查标题是否接近，现象描述、复现步骤是否一致。
- **证据相似度**：如果复现手法相同且页面结果一致（例如：相同页面元素在移动端被裁切），默认视为同一 Issue。只有当受影响页面、交互入口、错误表现或修复边界明显不同，才考虑拆分。
- **补充证据**：发现重复 Issue 但证据不同时，在原 Issue 补充证据，而非新建 Issue。

### 6. 提交 Issue

Issue 标题建议格式：`[UI/Bug/Crash] [问题简要描述]`

Issue 正文包含：
- 问题描述、复现步骤、预期行为、实际行为、严重程度
- **证据链接/附件**：所有 Issue 必须在正文或评论中补全对应的仓库内 Blob 链接或附件说明。

严重程度判断：
- 崩溃/白屏/无法继续操作 -> `Crash` 或 `Major`
- 功能错误但有替代路径 -> `Major`
- UI 异常、文案、间距、轻微交互 -> `Minor` 或 `Trivial`

## 输出要求

- **聚焦单一问题**：每轮对话仅产出一个 Issue，确保问题描述详尽、证据充分。
- 所有测试记录和 Issue 文案使用中文。
- 记录已测试区域，避免重复测试。
- 发现重复 Issue 时，明确说明是“判定为重复”还是“补充证据”。
- **强制提交**：发现新问题必须立即调用工具提交或补充证据。
