# Frontend Exploratory Testing

这个 skill 用于以前端工程师视角做探索性测试，目标不是“跑完脚本”，而是主动找出真实缺陷并形成可提交的 Issue。

## 适用场景

- 需要快速浏览并测试一个前端软件的主要功能
- 需要检查登录、表单、跳转、响应式布局、错误提示、控制台报错
- 需要把发现的问题整理成可直接提交到 Issue Tracker 的内容

## 核心原则

- 先观察，再操作；先定位重点区域，再深入测试
- 每次发现问题，先采集证据，再做查重
- 只查 `open` 状态的 Issue
- 发现重复问题时，不只比标题和描述，还要对比证据相似度
- 高度相似则跳过，继续找下一个问题
- 同问题但证据不相似，则补充当前证据后再提交
- 保持中文输出，Issue 内容要具体、可复现、可验证

## 工作流

### 1. 制定测试计划

开始前先快速浏览页面和主要入口，列出本轮重点区域，例如：

- 登录/注册流程
- 表单填写与提交
- 页面跳转与返回
- 空状态、错误状态、加载状态
- 响应式布局
- 控制台报错和网络失败

记录当前轮已经测试过的区域，避免重复覆盖同一路径。

### 2. 执行探索性测试

使用 `agent-browser` 与页面交互：

- 点击按钮、菜单、链接
- 输入正常值、边界值、非法值
- 切换页面、刷新、返回
- 缩放到不同分辨率或移动端宽度
- 观察 DOM、样式、遮挡、溢出、文案截断
- 查看控制台日志和网络请求失败

每次操作后都检查页面反馈是否符合预期，关注：

- UI 错位、遮挡、闪烁、溢出
- 点击无响应、状态未更新、数据不一致
- 错误提示缺失、文案不友好、恢复路径不清晰
- JS 报错、请求失败、白屏、卡死、崩溃

更完整的测试发散清单见 [exploration-playbook.md](references/exploration-playbook.md)。

### 3. 10 分钟换思路

如果在同一方向上持续约 10 分钟没有发现新问题，立即切换到另一个角度：

- 正常流程 -> 异常输入
- 桌面端 -> 移动端
- UI 检查 -> 网络层检查
- 单页检查 -> 跨页面流程
- 功能检查 -> 错误恢复与边界值

切换时记录原因和新的测试方向，避免在同一路径上反复耗时。

### 4. 发现问题后的证据采集

普通问题优先截图，截图要包含完整上下文。截图取证默认使用本地运行的客户端页面或本地 H5，不要改用线上测试站点作为截图来源。

崩溃、白屏、无响应问题优先录屏，同时保存：

- 崩溃前的具体操作顺序
- 页面状态变化
- 控制台完整报错栈

证据要足以让别人在相同环境中复现。

### 5. 查重与相似度判断

提交前，只检查 Issue Tracker 中 `open` 状态的 Issue。

查重时先看：

- 标题是否接近
- 现象描述是否一致
- 复现步骤是否一致
- 证据是否一致，尤其是截图、录屏、控制台报错、网络错误

如果多个问题使用同样的复现手法，并且得到同样的页面结果，例如都属于“移动端窄屏下元素被裁切、截断或超出视口”，且从用户角度看到的现象一致，那么默认应视为同一个 issue，不要拆成多个 issue。只有当受影响页面、交互入口、错误表现或修复边界明显不同，才考虑拆分。

更完整的证据相似度与查重判定规则见 [dedup-evidence.md](references/dedup-evidence.md)。

### 6. 提交 Issue

Issue 标题建议格式：

- `[UI] 登录按钮在移动端错位`
- `[Bug] 表单提交后状态未更新`
- `[Crash] 切换页面后白屏`

Issue 正文建议包含：

- 问题描述
- 复现步骤
- 预期行为
- 实际行为
- 截图或录屏
- 环境信息
- 控制台报错
- 严重程度
- 截图附件或可直接打开的截图链接；截图必须同步附在 Issue 正文或评论中，优先作为 Issue 附件上传，不能只保存在本地。私有仓库不要使用 `raw.githubusercontent.com` 作为唯一证据链接，优先使用 GitHub Issue 附件或仓库内可访问的 blob/raw 链接。若命令行工具无法直接上传附件，先将截图文件提交到临时分支，再用分支中的文件链接补到 Issue 评论中。

在完成查重后，只要确认是新问题或证据需要补充的新证据，就必须立即提交 Issue，不要只停留在本地记录或口头结论。

严重程度按实际影响判断：

- 崩溃 / 白屏 / 无法继续操作 -> `Crash` 或 `Major`
- 功能错误但有替代路径 -> `Major`
- UI 异常、文案、间距、轻微交互问题 -> `Minor` 或 `Trivial`

## 输出要求

- 所有测试记录和 Issue 文案都使用中文
- 记录已测试区域，避免重复测试
- 发现重复 Issue 时，要明确说明是“证据高度相似，判定为重复”还是“现象相同但证据不同，已补充证据”
- 不要放过小问题，但优先提交有实际影响、可复现、证据充分的问题
