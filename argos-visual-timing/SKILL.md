# Argos 视觉截图时机优化

这个 skill 只解决一类问题：视觉截图不是“拍不出来”，而是“拍得太晚、太早，或者拍到了重复态”。

## 什么时候用

当你看到下面任一情况时，优先使用这个 skill：

- Argos build 中同一页面态反复变化，但代码本身改动不大
- 一组截图里出现“入口态”和“稳定态”很像、但其实不是同一张图
- 页面已经有稳定 test id，却还在用固定 `waitForTimeout(...)`
- 需要决定某个页面该早截、晚截，还是继续默认等待

## 工作流

1. 先读最新 build diff，按截图名判断 changed / unchanged。
2. 把截图分成三类：
   - 早截：需要尽快抓入口态，避免后续 UI 覆盖
   - 晚截：需要等内容锚点稳定后再抓
   - 默认：现有等待已经够用
3. 对“晚截”页面，优先加页面锚点等待，不先加长固定 timeout。
4. 对“早截”页面，减少 spinner 等待和额外 settle。
5. 只在少数例外页上写特殊时机，其余页面尽量走默认配置。
6. 复验时只盯最敏感的 1 到 2 个页面，先确认时机变化没有把截图拍歪。

## 决策规则

- 入口页、登录入口、首次出现的壳层态：优先早截。
- 动态内容页、商城页、奖励页、需要等卡片/列表/标签稳定的页面：优先晚一点截。
- 如果页面已经有明确 test id 可等到出现，就不要继续堆固定时间。
- 如果同一页面有多个相近截图，优先合并或拉开时机，而不是继续新增近似图。

## 推荐做法

- 用截图名维护一个小型时机表，只给例外页单独配置。
- 在视觉测试里优先等待“真实内容锚点”：
  - 关键按钮可见
  - 列表项可见
  - 预览卡片可见
  - 页面切换后主内容可见
- 对早截页可跳过 spinner 等待，但要保留足够的页面就绪断言。
- 先稳定截图态，再考虑改业务页面锚点。
